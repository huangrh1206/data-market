# 实验设计

## 实验环境

实验系统基于 Hyperledger Fabric 测试网络部署，链码实现 Q-DataMarket 的数据产品、订单、授权、质量评分和仲裁逻辑。链下数据采用本地对象存储或 MinIO 保存，系统只在链上保存数据摘要、元数据哈希、访问凭证哈希、质量评分和争议证据。性能测试使用 Hyperledger Caliper 发起工作负载。

IoT 数据采用 CSV/JSON 批次时序数据。每个数据产品对应一个设备在一个时间窗口内产生的数据文件。实验数据包括高质量、中质量和低质量三类，通过注入缺失字段、异常值、时间戳断裂和非法状态码构造不同质量等级。

## 实验一：系统基础性能测试

### 目的

验证 Q-DataMarket 在不同交易负载下的吞吐能力、延迟表现和交易成功率，说明链上链下协同架构的工程可行性。

### 交易类型

```text
RegisterProduct
CreateOrder
IssueAccessToken
ConfirmDelivery
SubmitRating
OpenDispute
ResolveDispute
```

### 变量设置

```text
并发用户数：50、100、200、500
交易比例：发布 20%，购买 40%，授权 15%，确认 15%，评价 8%，争议 2%
数据规模：每个批次 120 条、600 条、1200 条记录
```

### 评价指标

```text
TPS
平均延迟
P95 延迟
交易成功率
CPU 使用率
内存占用
网络 IO
账本大小增长
```

### 分析口径

若并发升高时 TPS 先提升后趋于稳定，平均延迟和 P95 延迟随负载增加而上升，则说明系统性能受 Fabric 背书、排序和提交流程约束。通过比较不同交易类型延迟，可以分析仲裁和评价交易相对普通订单交易的额外开销。

## 实验二：质量驱动机制有效性测试

### 目的

验证质量评分和卖家信誉是否能提升高质量数据成交概率，并降低争议率。

### 对比方案

```text
Baseline：随机排序，固定价格，不使用质量评分和卖家信誉。
Q-Score：按照质量评分和卖家信誉联合排序，并根据质量分调整推荐价格。
```

### 数据设置

构造高、中、低质量数据产品各 100 个。高质量数据缺失率和异常率较低，中质量数据含少量缺失或时间戳不连续，低质量数据含明显异常值、缺失字段和非法状态码。

### 评价指标

```text
高质量数据成交占比
平均成交质量分
买家满意度评分
争议率
卖家信誉变化
```

### 分析口径

若 Q-Score 方案相较 Baseline 具有更高的高质量数据成交占比、更高的平均成交质量分和更低的争议率，则说明质量评分和信誉排序机制能够改善交易匹配质量。

## 实验三：押金仲裁机制有效性测试

### 目的

验证押金锁定、证据记录和状态机仲裁对异常交易的识别与约束能力。

### 异常场景

```text
场景 1：卖家上传数据与链上 dataHash 不一致。
场景 2：买家下载成功但超时不确认。
场景 3：买家发起虚假争议。
场景 4：访问凭证过期后重复访问链下数据。
```

### 评价指标

```text
异常交易识别率
平均仲裁延迟
恶意行为损失成本
正常交易成功率
押金扣罚准确率
```

### 分析口径

若系统能够对哈希不一致、超时不确认和虚假争议执行正确的退款、结算或扣罚，并保持较高正常交易成功率，则说明押金仲裁机制能够在不引入中心化平台完全控制的情况下提升交易公平性。
